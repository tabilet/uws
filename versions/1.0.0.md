# Udon Workflow Specification v1.0.0

## Version 1.0.0

This document defines the Udon Workflow Specification (UWS) 1.0.0.

The key words `MUST`, `MUST NOT`, `REQUIRED`, `SHALL`, `SHALL NOT`, `SHOULD`, `SHOULD NOT`, `RECOMMENDED`, `NOT RECOMMENDED`, `MAY`, and `OPTIONAL` in this document are to be interpreted as described in BCP 14 when, and only when, they appear in all capitals.

The JSON Schema for UWS 1.0 documents is identified by:

```text
https://spec.udon-workflow.org/1.0/schema.json
```

This identifier is the schema `$id`. Until a canonical hosted copy is available, implementations MAY resolve it by fetching the `uws.json` file from the UWS repository.

## What is the UWS Specification?

The UWS Specification defines a compact workflow overlay for OpenAPI-backed HTTP operations. It describes the operation bindings, request values, dependencies, structural control flow, triggers, and runtime expression fields needed by workflow tooling while leaving HTTP operation details in OpenAPI.

## Status of This Document

This document describes UWS 1.0.0. The repository schema and Go validation APIs are the implementation conformance targets for this package.

## 1. UWS Specification

### 1.1 Version 1.0.0

The `uws` field identifies the UWS version used by a UWS document. For this version family, the value MUST match:

```text
^1\.\d+\.\d+(-.+)?$
```

Examples include `1.0.0`, `1.1.0`, and `1.0.0-beta.1`.

Patch versions address corrections and clarifications for the 1.0 feature set.

## 2. Introduction

UWS defines a language-agnostic workflow document for orchestrating HTTP API operations described by OpenAPI.

UWS is intentionally not a replacement for OpenAPI. OpenAPI owns HTTP methods, paths, request and response schemas, servers, and security schemes. UWS owns the workflow overlay: operation binding, sequencing, request value binding, structural control flow, triggers, result extraction, and runtime expression fields.

UWS documents are serialized as JSON, YAML, or canonical HCL. They serve as an interchange format for workflow tooling, including the `udon` implementation, but the specification does not require a particular runtime.

HCL is an authoring format for `udon` workflows. A UWS document is a materialized workflow interchange document. Expression fields are serialized as strings, and structured bodies such as `request`, `variables`, and trigger `options` are represented as JSON/YAML-compatible values.

The Go `convert` package provides JSON, YAML, and HCL conversion helpers over UWS documents. JSON and YAML preserve `x-*` extensions through the extension model. HCL conversion is intended for canonical UWS fields and dynamic map bodies; documents with `x-*` extensions are rejected for HCL conversion because extension fields are not emitted as HCL attributes.

## 3. Definitions

### 3.1 UWS Document

A self-contained workflow document conforming to this specification.

### 3.2 Source Description

An OpenAPI document referenced by operations in a UWS document.

### 3.3 Operation

A UWS-local executable unit bound to one operation in a referenced OpenAPI document, or an extension-owned operation identified by an implementation profile.

### 3.4 Workflow

A structural control-flow construct that composes operations or nested steps into sequence, parallel, switch, merge, loop, or await patterns.

### 3.5 Step

A unit inside a workflow. A step either references an operation by `operationRef` or describes a structural workflow step through `type`.

### 3.6 Trigger

An entry point that initiates workflow execution, such as a webhook endpoint.

### 3.7 Runtime Expression

A string expression evaluated at execution time, used for conditional logic (`when`), iteration (`forEach`), waits (`wait`), workflow references (`workflow`), outputs (`outputs`), and success criteria (`successCriteria`).

### 3.8 Specification Extension

A custom field prefixed with `x-`. Extension fields are preserved by compliant tooling but are not interpreted by the UWS core specification.

## 4. Specification

### 4.1 Format

A UWS document is a JSON object. It MAY be represented as JSON, YAML, or canonical HCL.

All field names in the specification are case sensitive unless explicitly noted otherwise.

JSON and YAML documents MAY contain specification extensions. HCL conversion is limited to canonical UWS fields and dynamic map bodies.

### 4.2 Data Types

Data types in UWS are based on JSON-compatible values and JSON Schema-compatible schema objects.

The following type names are used in this document:

| Type | Meaning |
|---|---|
| `string` | JSON string. |
| `number` | JSON number. |
| `integer` | JSON number without a fraction or exponent part. |
| `boolean` | JSON boolean. |
| `any` | Any JSON-compatible value. |
| `Map[string, any]` | JSON object with string keys. |
| `ParamSchema` | UWS parameter schema object. |
| `[T]` | JSON array of values of type `T`. |

### 4.3 Relative References in URLs

Unless otherwise specified, URL fields MAY use relative references. Relative source description URLs are resolved relative to the containing UWS document. For in-memory documents, the implementation supplies the base directory or base URL.

### 4.4 Relationship To OpenAPI

Every executable core operation MUST bind to an OpenAPI operation through a `sourceDescription` plus exactly one of:

- `openapiOperationId`: the OpenAPI `operationId`.
- `openapiOperationRef`: a JSON Pointer fragment to an OpenAPI operation when `operationId` is unavailable or unsuitable.

UWS core MUST NOT duplicate OpenAPI method, path, schema, server, or security data. Tools that need those details derive them from the referenced OpenAPI document.

Credential storage, provider defaults, non-HTTP runtimes, and product-specific execution metadata belong in `x-*` extensions or implementation profiles. An operation without an OpenAPI binding is an extension-owned operation and MUST include `x-uws-operation-profile` with at least one non-whitespace character to name the implementation profile that can execute it.

HTTP method, path, server, request/response schema, and security information are resolved from the referenced OpenAPI operation, not from UWS core fields.

`openapiOperationRef` is a JSON Pointer fragment resolved against the OpenAPI document named by `sourceDescription`. It MUST begin with `#/`. UWS core does not load the referenced OpenAPI document; resolution of the pointer target is the responsibility of the execution runtime.

If `sourceDescription`, `openapiOperationId`, and `openapiOperationRef` are all omitted, the operation is extension-owned and MUST include `x-uws-operation-profile`. Extension-owned operations are valid UWS documents but are executable only by implementations that understand the selected extension profile.


##### 4.5.6.3 Workflow Type Semantics

The `type` field selects one of six structural constructs. The following definitions apply to both Workflow objects and Step objects that set `type`.

- **`sequence`** — Steps execute in declaration order. Each step MUST complete before the next begins. `dependsOn` MAY add cross-workflow dependencies that extend the declaration order. `items` MUST NOT be set on a `sequence`.

- **`parallel`** — Steps execute concurrently, subject only to `dependsOn` and `parallelGroup` relationships. The construct completes when every step has terminated. `parallelGroup` is an identifier that names a shared barrier scope for coordination across sibling constructs; it is informational for UWS core and its scheduling semantics are implementation-defined. `items` MUST NOT be set on a `parallel`.

- **`switch`** — Exactly one `case` whose `when` expression evaluates truthy executes its `steps`. If no `case` matches and `default` is present, `default` executes. If no `case` matches and `default` is absent, the construct emits no result. `items` MUST NOT be set on a `switch`.

- **`loop`** — Iterates over a JSON array. `items` is REQUIRED and MUST be a runtime expression that resolves to a JSON array. Each element is bound into the iteration scope and made available to expressions within `steps`. If `batchSize` is set, it MUST be a runtime expression that resolves to a positive integer; iterations are partitioned into fixed-size batches executed sequentially. `cases` and `default` MUST NOT be set on a `loop`.

- **`merge`** — Combines the outputs of multiple constructs named by `dependsOn` into a single structural result. `dependsOn` is REQUIRED on a `merge` construct and MUST name at least one upstream construct. The result is declared by a Structural Result Object whose `kind` is `merge` and whose `from` names the merge construct (see §4.5.14). `mode` is implementation-defined but, when present, MUST be a string. `items` MUST NOT be set on a `merge`.

- **`await`** — Blocks execution until its `wait` expression evaluates truthy, or until an implementation-defined timeout elapses. `wait` is REQUIRED on an `await`. `cases`, `default`, and `items` MUST NOT be set on an `await`.

### 4.6 Validation

Core validation MUST check:

- `uws`, `info`, and `operations` are present.
- Version strings match the UWS 1.x pattern.
- Source description names, operation IDs, workflow IDs, step IDs, trigger IDs, and parallel groups are unique where applicable.
- Workflow IDs and step IDs match `^[A-Za-z0-9_-]+$` so `StructuralResult.from` remains unambiguous.
- Source descriptions are OpenAPI sources.
- Each OpenAPI-bound operation references an existing source description.
- Each OpenAPI-bound operation sets exactly one of `openapiOperationId` or `openapiOperationRef`.
- Each extension-owned operation includes `x-uws-operation-profile` with at least one non-whitespace character.
- Dependency, operation, workflow, step, trigger route, success action, and failure action references resolve.
- Structural workflow and step types are known.
- Structural construct field constraints hold (see §4.5.6.3): `loop` requires `items`; `await` requires `wait`; `switch`, `sequence`, `parallel`, `merge`, and `await` MUST NOT set `items`; `loop` and `await` MUST NOT set `cases` or `default`.
- Each `StructuralResult` resolves `from` to a known workflow or `workflowId.stepId` pair whose `type` is structural and matches the result `kind` (see §4.5.14).
- Each trigger route `output` references a declared entry in the parent trigger's `outputs` list (see §4.5.9).
- `x-*` fields are preserved as specification extensions.

Schema validation checks document shape. Semantic checks such as duplicate identifiers and reference integrity are enforced by the Go `Validate` and `ValidateResult` APIs.

## 5. Runtime Expressions

UWS uses runtime expression strings in control-flow fields (`when`, `forEach`, `wait`, `workflow`, `items`, `batchSize`), in criterion conditions, and in the string values of `outputs` maps. A runtime expression is a string whose resolved value is a JSON-compatible value determined at execution time.

Conforming implementations MUST accept the expression sources defined in §5.1, the JSON Pointer fragment form in §5.2, the output dot-walk in §5.3, and the comparison operators in §5.4. Implementations MAY support additional syntax as described in §5.5.

### 5.1 Expression Sources

Expression strings reference data through these normative sources. Each source begins with a leading `$` sigil. A dotted suffix `.<path>` is a sequence of one or more dot-separated segments, where each segment matches `[a-zA-Z0-9._-]+` and is resolved against the current value as an object property name or array index.

| Source | Meaning |
|---|---|
| `$response.statusCode` | HTTP response status code of the current operation. |
| `$response.body` | HTTP response body of the current operation. |
| `$response.body<json-pointer>` | A JSON Pointer fragment (§5.2) into the response body. |
| `$response.headers.<name>` | HTTP response header value by name. |
| `$outputs.<name>` | Same-scope output value declared by the enclosing operation, workflow, or step. |
| `$outputs.<name>.<path>` | Dot-walk into a structured output value (see §5.3). |
| `$steps.<stepId>.outputs.<name>` | Output of a named sibling step within the same workflow. |
| `$steps.<stepId>.outputs.<name>.<path>` | Dot-walk into a named sibling step's output value. |
| `$variables.<name>` | A document-scope variable declared under `variables` or `components.variables`. |
| `$variables.<name>.<path>` | Dot-walk into a structured variable value. |
| `$trigger` | The input payload delivered by the enclosing trigger. |
| `$trigger.<path>` | Dot-walk into the trigger payload. |

A resolved value MAY be of any JSON type: string, number, boolean, null, object, or array.

### 5.2 JSON Pointer Fragments

A JSON Pointer fragment `<json-pointer>` is an RFC 6901 JSON Pointer prefixed with `#`. It MAY be used as a suffix on `$response.body` and as the value of `Criterion.context` when `Criterion.type` is `jsonpath`. The pointer resolves against the value named by its prefix.

Example:

```text
$response.body#/items/0/name
```

### 5.3 Output Values and Dot-Walk

Each entry in an `outputs` map of an operation, workflow, or step is a pair `<name>: <expression>`. The `<expression>` is a string that conforms to §5.1. At execution time, each entry is evaluated and the resolved value is bound to `<name>` in the enclosing scope.

A resolved output value MAY be of any JSON type. Consumers MAY dot-walk into structured outputs:

- `$outputs.<name>` resolves to the whole bound value.
- `$outputs.<name>.<segment>` resolves to the result of applying `<segment>` as an object property name (for JSON objects) or a decimal array index (for JSON arrays).
- Subsequent segments apply recursively.

If a segment does not resolve — the property is absent from a JSON object, the index is out of bounds on a JSON array, or the segment is applied to a scalar — the expression value is `null`.

The typed shape `Map[string, string]` in the UWS document model reflects the *expression* type, not the resolved runtime type.

### 5.4 Comparison Operators

Inside `when`, `Criterion.condition` when `Criterion.type` is `simple`, and any field documented as a boolean expression, conforming implementations MUST support the comparison operators `==`, `!=`, `<`, `<=`, `>`, and `>=`.

- Operands MUST be of the same JSON type. Implementations MUST NOT perform implicit type coercion.
- `==` and `!=` are defined for strings, numbers, booleans, and `null`.
- `<`, `<=`, `>`, and `>=` are defined for strings (lexicographic) and numbers. Applying an ordering operator to booleans, `null`, objects, or arrays is an error; implementations MUST treat the comparison as failing.

### 5.5 Implementation Extensions

Richer expression features — such as boolean connectives (`&&`, `||`, `!`), arithmetic, function calls (for example `length()`), or string operations — are implementation-defined. A conforming UWS core implementation MUST NOT require such features in order to parse documents that use only §5.1–§5.4 syntax. A UWS document that depends on features beyond §5.1–§5.4 is tied to the implementation profile that defines them and SHOULD declare that dependency via an `x-uws-operation-profile` or equivalent extension marker on the relevant operation.

## 6. Specification Extensions

Specification extensions are fields whose names begin with `x-`.

UWS core does not define service types for local commands, functions, file I/O, SSH, SMTP, SQL, object storage, LLM calls, or other non-OpenAPI behavior. These capabilities can be described through `x-*` extension profiles.

Example non-normative extension:

```json
{
  "operationId": "build_email",
  "x-uws-operation-profile": "udon",
  "x-udon-runtime": {
    "type": "fnct",
    "function": "mail_raw",
    "args": {
      "subject": "Daily weather report"
    }
  }
}
```

The core specification preserves extension fields but does not validate or execute implementation-specific extension payloads.

## 7. Security Considerations

UWS documents can contain request values, runtime expressions, trigger definitions, and implementation extensions. Tooling SHOULD treat documents from untrusted sources as untrusted input.

UWS core does not define credential storage. Credentials, provider defaults, and runtime-specific authentication metadata belong in implementation profiles or external configuration.

OpenAPI-bound operation security requirements are derived from referenced OpenAPI documents. UWS tooling SHOULD validate that referenced source descriptions are the intended documents before execution.
