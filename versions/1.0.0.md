# Udon Workflow Specification v1.0.0

## Introduction

The Udon Workflow Specification (UWS) defines a language-agnostic workflow document for orchestrating HTTP API operations that are described by OpenAPI.

UWS is intentionally not a replacement for OpenAPI. OpenAPI owns HTTP methods, paths, request and response schemas, servers, and security schemes. UWS owns the workflow overlay: operation sequencing, request value binding, control flow, triggers, result extraction, and runtime expressions.

UWS documents are serialized as JSON, YAML, or canonical HCL. They serve as an interchange format for workflow tooling, including the `udon` implementation, but the specification does not require a particular runtime.

The JSON Schema for UWS 1.0 documents is available at:

```text
https://github.com/tabilet/uws/1.0/schema
```

## Definitions

### UWS Document

A self-contained workflow document conforming to this specification.

### Source Description

An OpenAPI document referenced by operations in the UWS document.

### Operation

A UWS-local executable unit bound to one operation in a referenced OpenAPI document.

### Workflow

A structural control-flow construct that composes operations or nested steps into sequence, parallel, switch, merge, loop, or await patterns.

### Step

A unit inside a workflow. A step either references an operation by `operationRef` or describes a structural workflow step through `type`.

### Trigger

An entry point that initiates workflow execution, such as a webhook endpoint.

### Runtime Expression

A string expression evaluated at execution time, used for conditional logic (`when`), iteration (`forEach`), waits (`wait`), workflow references (`workflow`), outputs (`outputs`), and success criteria (`successCriteria`).

### Specification Extension

A custom field prefixed with `x-`. Extension fields are preserved by compliant tooling but are not interpreted by the UWS core specification.

## Version

The `uws` field specifies the UWS version. For this version, the value MUST match:

```text
^1\.\d+\.\d+(-.+)?$
```

Examples: `1.0.0`, `1.1.0`, `1.0.0-beta.1`.

## Relationship To OpenAPI

Every executable core operation MUST bind to an OpenAPI operation through a `sourceDescription` plus exactly one of:

- `openapiOperationId`: the OpenAPI `operationId`.
- `openapiOperationRef`: a JSON Pointer fragment to an OpenAPI operation when `operationId` is unavailable or unsuitable.

UWS core MUST NOT duplicate OpenAPI method, path, schema, server, or security data. Tools that need those details derive them from the referenced OpenAPI document.

Credential storage, provider defaults, non-HTTP runtimes, and product-specific execution metadata belong in `x-*` extensions or implementation profiles. An operation without an OpenAPI binding is an extension-owned operation and MUST include `x-uws-operation-profile` to name the implementation profile that can execute it.

## Relationship To HCL

HCL is a canonical authoring format for `udon` workflows. HCL authoring can contain expressions, variables, function calls, conditional logic, and `for` expressions that are not directly representable as plain JSON/YAML values.

A UWS document is a materialized workflow interchange document. Expression fields are serialized as strings. Structured bodies such as `request`, `variables`, and trigger `options` are represented as JSON/YAML-compatible values.

The Go `convert` package provides JSON, YAML, and HCL conversion helpers over UWS documents. JSON and YAML preserve `x-*` extensions through the extension model. HCL conversion is intended for canonical UWS fields and dynamic map bodies; documents with `x-*` extensions are rejected for HCL conversion because extension fields are not emitted as HCL attributes.

## Document Structure

A UWS document MUST contain:

| Field | Type | Description |
|---|---|---|
| `uws` | `string` | UWS version string. |
| `info` | [Info Object](#info-object) | Document metadata. |
| `operations` | \[[Operation Object](#operation-object)\] | Workflow operations. MUST contain at least one item. |

Optional top-level fields:

| Field | Type | Description |
|---|---|---|
| `variables` | `Map[string, any]` | Global variables available to expressions. |
| `sourceDescriptions` | \[[Source Description Object](#source-description-object)\] | OpenAPI source documents. Required when any operation is OpenAPI-bound. |
| `workflows` | \[[Workflow Object](#workflow-object)\] | Structural workflow definitions. |
| `triggers` | \[[Trigger Object](#trigger-object)\] | Workflow entry points. |
| `results` | \[[Structural Result Object](#structural-result-object)\] | Outputs from structural constructs. |
| `components` | [Components Object](#components-object) | Reusable UWS-local values. |
| `x-*` | `any` | Specification extensions. |

### Document Example

```json
{
  "uws": "1.0.0",
  "info": {
    "title": "Pet Store Workflow",
    "version": "1.0.0"
  },
  "sourceDescriptions": [
    {
      "name": "petstore_api",
      "url": "./petstore.yaml",
      "type": "openapi"
    }
  ],
  "operations": [
    {
      "operationId": "list_pets",
      "sourceDescription": "petstore_api",
      "openapiOperationId": "listPets",
      "request": {
        "query": {
          "limit": 10
        }
      },
      "outputs": {
        "firstPet": "$response.body#/0"
      }
    }
  ]
}
```

## Objects

### Info Object

| Field | Type | Required | Description |
|---|---|---|---|
| `title` | `string` | yes | Human-readable title. |
| `version` | `string` | yes | Document version distinct from the UWS spec version. |
| `summary` | `string` | no | Short summary. |
| `description` | `string` | no | Longer CommonMark-capable description. |
| `x-*` | `any` | no | Extensions. |

### Source Description Object

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | `string` | yes | Unique source name. MUST match `^[A-Za-z0-9_\-]+$`. |
| `url` | `string` | yes | URL or relative path to an OpenAPI document. |
| `type` | `string` | no | When present, MUST be `openapi`. |
| `x-*` | `any` | no | Extensions. |

Relative `url` values are resolved relative to the containing UWS document. For in-memory documents, the implementation supplies the base directory or URL.

### Operation Object

An operation is a UWS-local executable unit bound to one OpenAPI operation. `operationId` is local to the UWS document; it does not have to equal the OpenAPI `operationId`.

| Field | Type | Required | Description |
|---|---|---|---|
| `operationId` | `string` | yes | Unique UWS-local operation identifier. |
| `sourceDescription` | `string` | conditional | Name of the OpenAPI source description. Required for OpenAPI-bound operations. |
| `openapiOperationId` | `string` | conditional | OpenAPI `operationId`. Exactly one of this or `openapiOperationRef` is required for OpenAPI-bound operations. |
| `openapiOperationRef` | `string` | conditional | JSON Pointer fragment to an OpenAPI operation. Exactly one of this or `openapiOperationId` is required for OpenAPI-bound operations. |
| `description` | `string` | no | Operation description. |
| `request` | [Request Binding Object](#request-binding-object) | no | Request values to bind into the OpenAPI operation. |
| `dependsOn` | `[string]` | no | Operations, workflows, steps, or parallel groups that must complete first. |
| `when` | `string` | no | Conditional execution expression. |
| `forEach` | `string` | no | Iteration expression. |
| `wait` | `string` | no | Delay expression. |
| `workflow` | `string` | no | Sub-workflow path or runtime reference. |
| `parallelGroup` | `string` | no | Parallel group identifier. |
| `successCriteria` | \[[Criterion Object](#criterion-object)\] | no | Success conditions. |
| `onFailure` | \[[Failure Action Object](#failure-action-object)\] | no | Failure handlers. |
| `onSuccess` | \[[Success Action Object](#success-action-object)\] | no | Success handlers. |
| `outputs` | `Map[string, string]` | no | Named runtime expression outputs. |
| `x-uws-operation-profile` | `string` | conditional | Required for extension-owned operations; names the non-whitespace implementation profile that can execute the operation. |
| `x-*` | `any` | no | Extensions. |

HTTP method, path, server, request/response schema, and security information are resolved from the referenced OpenAPI operation, not from UWS core fields.

If `sourceDescription`, `openapiOperationId`, and `openapiOperationRef` are all omitted, the operation is extension-owned and MUST include `x-uws-operation-profile` with at least one non-whitespace character. Extension-owned operations are valid UWS documents but are executable only by implementations that understand the selected extension profile.

| Field | Type | Required | Description |

| `header` | `Map[string, any]` | no | Values for OpenAPI header parameters. |
| `cookie` | `Map[string, any]` | no | Values for OpenAPI cookie parameters. |
| `body` | `any` | no | Request body value. |
| `x-*` | `any` | no | Request-binding extensions. |

## Runtime Expressions

UWS uses runtime expression strings in control-flow and output fields. The exact evaluator is implementation-defined, but expressions commonly reference the current operation response, prior operation outputs, variables, and trigger input.

Common expression contexts include:

| Expression | Meaning |
|---|---|
| `$response.statusCode` | HTTP response status code from the current operation. |
| `$response.body` | HTTP response body from the current operation. |
| `$response.headers` | HTTP response headers from the current operation. |
| `$outputs.<name>` | Named output value. |
| `$trigger` | Trigger input data. |

Implementations MAY support richer expression syntax. If they do, that behavior belongs to the implementation profile, not the UWS core schema.

## Extension Runtime Profiles

UWS core does not define service types for local commands, functions, file I/O, SSH, SMTP, SQL, object storage, LLM calls, or other non-OpenAPI behavior. These capabilities can be described through `x-*` extension profiles.

Example non-normative extension:

```json
{
  "operationId": "build_email",
  "x-uws-operation-profile": "udon",
  "x-udon-runtime": {
    "type": "fnct",
    "function": "mail_raw",
    "args": {
      "subject": "Daily weather report"
    }
  }
}
```

The core specification preserves this extension field but does not validate or execute it.

## Validation

Core validation MUST check:

- `uws`, `info`, and `operations` are present.
- Version strings match the UWS 1.x pattern.
- Source description names, operation IDs, workflow IDs, step IDs, trigger IDs, and parallel groups are unique where applicable.
- Source descriptions are OpenAPI sources.
- Each OpenAPI-bound operation references an existing source description.
- Each OpenAPI-bound operation sets exactly one of `openapiOperationId` or `openapiOperationRef`.
- Each extension-owned operation includes `x-uws-operation-profile` with at least one non-whitespace character.
- Dependency, operation, workflow, step, trigger route, success action, and failure action references resolve.
- Structural workflow and step types are known.
- `x-*` fields are preserved as specification extensions.

Schema validation checks document shape. Semantic checks such as duplicate identifiers and reference integrity are enforced by the Go `Validate` and `ValidateResult` APIs.
