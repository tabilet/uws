# Udon Workflow Specification v1.0.0

## Version 1.0.0

This document defines the Udon Workflow Specification (UWS) 1.0.0.

The key words `MUST`, `MUST NOT`, `REQUIRED`, `SHALL`, `SHALL NOT`, `SHOULD`, `SHOULD NOT`, `RECOMMENDED`, `NOT RECOMMENDED`, `MAY`, and `OPTIONAL` in this document are to be interpreted as described in BCP 14 when, and only when, they appear in all capitals.

The JSON Schema for UWS 1.0 documents is identified by:

```text
https://spec.udon-workflow.org/1.0/schema.json
```

This identifier is the schema `$id`. Until a canonical hosted copy is available, implementations MAY resolve it by fetching the `uws.json` file from the UWS repository.

## What is the UWS Specification?

The UWS Specification defines a compact workflow overlay for OpenAPI-backed HTTP operations. It describes the operation bindings, request values, dependencies, structural control flow, triggers, and runtime expression fields needed by workflow tooling while leaving HTTP operation details in OpenAPI.

## Status of This Document

This document describes UWS 1.0.0. The repository schema and Go validation APIs are the implementation conformance targets for this package.

## 1. UWS Specification

### 1.1 Version 1.0.0

The `uws` field identifies the UWS version used by a UWS document. For this version family, the value MUST match:

```text
^1\.\d+\.\d+(-.+)?$
```

Examples include `1.0.0`, `1.1.0`, and `1.0.0-beta.1`.

Patch versions address corrections and clarifications for the 1.0 feature set.

## 2. Introduction

UWS defines a language-agnostic workflow document for orchestrating HTTP API operations described by OpenAPI.

UWS is intentionally not a replacement for OpenAPI. OpenAPI owns HTTP methods, paths, request and response schemas, servers, and security schemes. UWS owns the workflow overlay: operation binding, sequencing, request value binding, structural control flow, triggers, result extraction, and runtime expression fields.

UWS documents are serialized as JSON, YAML, or canonical HCL. They serve as an interchange format for workflow tooling, including the `udon` implementation, but the specification does not require a particular runtime.

HCL is an authoring format for `udon` workflows. A UWS document is a materialized workflow interchange document. Expression fields are serialized as strings, and structured bodies such as `request`, `variables`, and trigger `options` are represented as JSON/YAML-compatible values.

The Go `convert` package provides JSON, YAML, and HCL conversion helpers over UWS documents. JSON and YAML preserve `x-*` extensions through the extension model. HCL conversion is intended for canonical UWS fields and dynamic map bodies; documents with `x-*` extensions are rejected for HCL conversion because extension fields are not emitted as HCL attributes.

### 2.1 Scope and Relationship to Arazzo

UWS 1.0 intentionally narrows scope relative to the OpenAPI Initiative Arazzo specification. The two documents describe workflows over API operations defined outside the workflow document, but they target different tooling contracts and use different object models. A detailed side-by-side comparison is maintained at `versions/arazzo.md`.

UWS core does NOT define:

- A reusable-component registry for criteria, success actions, or failure actions. Criteria and actions are declared inline on the operation or workflow that uses them. Cross-operation reuse is expressed through implementation profiles or by duplicating the declaration.
- A separate Parameter object or a Reusable Object (`{reference, value}`). Request values are carried by the Request Binding Object (§4.5.5) whose keys map directly to OpenAPI parameter locations (`path`, `query`, `header`, `cookie`, `body`).
- A payload-replacement model for request bodies. The body value is bound directly under `request.body`.
- Source documents other than OpenAPI. Non-HTTP runtimes are expressed through extension-owned operations (§4.5.4) carrying `x-uws-operation-profile`.

Reuse of runtime behavior is delegated to the referenced OpenAPI document (for HTTP operations) and to implementation profiles (for extension-owned operations). A reader coming from Arazzo should expect a smaller surface and a stricter binding contract rather than a re-implementation of Arazzo's reuse model.

## 3. Definitions

### 3.1 UWS Document

A self-contained workflow document conforming to this specification.

### 3.2 Source Description

An OpenAPI document referenced by operations in a UWS document.

### 3.3 Operation

A UWS-local executable unit bound to one operation in a referenced OpenAPI document, or an extension-owned operation identified by an implementation profile.

### 3.4 Workflow

A structural control-flow construct that composes operations or nested steps into sequence, parallel, switch, merge, loop, or await patterns.

### 3.5 Step

A unit inside a workflow. A step either references an operation by `operationRef` or describes a structural workflow step through `type`.

### 3.6 Trigger

An entry point that initiates workflow execution, such as a webhook endpoint.

### 3.7 Runtime Expression

A string expression evaluated at execution time, used for conditional logic (`when`), iteration (`forEach`), waits (`wait`), workflow references (`workflow`), outputs (`outputs`), and success criteria (`successCriteria`).

### 3.8 Specification Extension

A custom field prefixed with `x-`. Extension fields are preserved by compliant tooling but are not interpreted by the UWS core specification.

### 3.9 Orchestrator

The UWS core component that manages workflow state transitions and structural control flow according to the specification.

### 3.10 Runtime Profile

An implementation-specific specialization that handles leaf operation execution and expression evaluation.

## 4. Specification

### 4.1 Format

A UWS document is a JSON object. It MAY be represented as JSON, YAML, or canonical HCL.

All field names in the specification are case sensitive unless explicitly noted otherwise.

JSON and YAML documents MAY contain specification extensions. HCL conversion is limited to canonical UWS fields and dynamic map bodies.

### 4.2 Data Types

Data types in UWS are based on JSON-compatible values and JSON Schema-compatible schema objects.

The following type names are used in this document:

| Type | Meaning |
|---|---|
| `string` | JSON string. |
| `number` | JSON number. |
| `integer` | JSON number without a fraction or exponent part. |
| `boolean` | JSON boolean. |
| `any` | Any JSON-compatible value. |
| `Map[string, any]` | JSON object with string keys. |
| `ParamSchema` | UWS parameter schema object. |
| `[T]` | JSON array of values of type `T`. |

### 4.3 Relative References in URLs

Unless otherwise specified, URL fields MAY use relative references. Relative source description URLs are resolved relative to the containing UWS document. For in-memory documents, the implementation supplies the base directory or base URL.

### 4.4 Relationship To OpenAPI

Every executable core operation MUST bind to an OpenAPI operation through a `sourceDescription` plus exactly one of:

- `openapiOperationId`: the OpenAPI `operationId`.
- `openapiOperationRef`: a JSON Pointer fragment to an OpenAPI operation when `operationId` is unavailable or unsuitable.

UWS core MUST NOT duplicate OpenAPI method, path, schema, server, or security data. Tools that need those details derive them from the referenced OpenAPI document.

Credential storage, provider defaults, non-HTTP runtimes, and product-specific execution metadata belong in `x-*` extensions or implementation profiles. An operation without an OpenAPI binding is an extension-owned operation and MUST include `x-uws-operation-profile` with at least one non-whitespace character to name the implementation profile that can execute it.

### 4.5 Schema

In the following object definitions, any field not marked `REQUIRED` is OPTIONAL unless a field description states a conditional requirement.

#### 4.5.1 UWS Document Object

This is the root object of a UWS document.

##### 4.5.1.1 Fixed Fields

| Field Name | Type | Description |
|---|---|---|
| `uws` | `string` | REQUIRED. UWS version string. |
| `info` | [Info Object](#452-info-object) | REQUIRED. Document metadata. |
| `operations` | [[Operation Object](#454-operation-object)] | REQUIRED. Workflow operations. The list MUST contain at least one item. |
| `variables` | `Map[string, any]` | Global variables available to runtime expressions. The value is an open-shape JSON object: keys and nested values are not restricted by UWS. |
| `sourceDescriptions` | [[Source Description Object](#453-source-description-object)] | OpenAPI source documents. REQUIRED when any operation is OpenAPI-bound. |

##### 4.5.1.2 UWS Document Object Example

```json

  "info": {
    "title": "Pet Store Workflow",
    "version": "1.0.0"

      "name": "petstore_api",
      "url": "./petstore.yaml",
      "type": "openapi"
    }
  ],
  "operations": [
    {
      "operationId": "list_pets",
      "sourceDescription": "petstore_api",
      "openapiOperationId": "listPets",
      "request": {
        "query": {
          "limit": 10
        }
      },
      "outputs": {
        "firstPet": "$response.body#/0"
      }
    }
  ]
}
```

#### 4.5.4 Operation Object

An operation is a UWS-local executable unit. `operationId` is local to the UWS document; it does not have to equal the OpenAPI `operationId`.

##### 4.5.4.1 Fixed Fields

| Field Name | Type | Description |
|---|---|---|
| `operationId` | `string` | REQUIRED. Unique UWS-local operation identifier. |
| `sourceDescription` | `string` | Name of the OpenAPI source description. REQUIRED for OpenAPI-bound operations. |
| `openapiOperationId` | `string` | OpenAPI `operationId`. Exactly one of this field or `openapiOperationRef` is REQUIRED for OpenAPI-bound operations. |
| `openapiOperationRef` | `string` | JSON Pointer fragment to an OpenAPI operation. Exactly one of this field or `openapiOperationId` is REQUIRED for OpenAPI-bound operations. |
| `description` | `string` | Operation description. |
| `request` | [Request Binding Object](#455-request-binding-object) | Request values to bind into the OpenAPI operation. |
| `dependsOn` | `[string]` | Operations, workflows, steps, or parallel groups that must complete first. |
| `when` | `string` | Conditional execution expression. |
| `forEach` | `string` | Iteration expression. |
| `wait` | `string` | Delay expression. |
| `parallelGroup` | `string` | Parallel dependency-barrier identifier. A `dependsOn` reference to the group waits for every member. |
| `successCriteria` | [[Criterion Object](#4511-criterion-object)] | Success conditions. |
| `onFailure` | [[Failure Action Object](#4512-failure-action-object)] | Failure handlers. |
| `onSuccess` | [[Success Action Object](#4513-success-action-object)] | Success handlers. |
| `outputs` | `Map[string, string]` | Named runtime expression outputs. Each value is a runtime expression string; the resolved value MAY be of any JSON type and MAY be dot-walked by consumers as described in §5.3. |
| `x-uws-operation-profile` | `string` | REQUIRED for extension-owned operations. The value MUST contain at least one non-whitespace character. |

This object MAY be extended with specification extensions.

HTTP method, path, server, request/response schema, and security information are resolved from the referenced OpenAPI operation, not from UWS core fields.

`openapiOperationRef` is a JSON Pointer fragment resolved against the OpenAPI document named by `sourceDescription`. It MUST begin with `#/`. UWS core does not load the referenced OpenAPI document; resolution of the pointer target is the responsibility of the execution runtime.

If `sourceDescription`, `openapiOperationId`, and `openapiOperationRef` are all omitted, the operation is extension-owned and MUST include `x-uws-operation-profile`. Extension-owned operations are valid UWS documents but are executable only by implementations that understand the selected extension profile.

Every operation MUST match exactly one of three mutually exclusive shapes. Documents that mix fields from two shapes or that omit the binding fields of every shape are invalid.

| # | Shape | `sourceDescription` | `openapiOperationId` | `openapiOperationRef` | `x-uws-operation-profile` |
|---|---|---|---|---|---|
| 1 | OpenAPI-bound by `operationId` | REQUIRED | REQUIRED | MUST NOT be set | OPTIONAL |
| 2 | OpenAPI-bound by JSON Pointer | REQUIRED | MUST NOT be set | REQUIRED | OPTIONAL |
| 3 | Extension-owned | MUST NOT be set | MUST NOT be set | MUST NOT be set | REQUIRED |

##### 4.5.4.2 Operation Object Example

```yaml
operationId: list_pets
sourceDescription: petstore_api
openapiOperationId: listPets
request:
  query:
    limit: 10
outputs:
  firstPet: $response.body#/0
```


##### 4.5.6.3 Workflow Type Semantics

The `type` field selects one of six structural constructs. The following definitions apply to both Workflow objects and Step objects that set `type`.

- **`sequence`** — Steps execute in declaration order. Each step MUST complete before the next begins. `dependsOn` MAY add cross-workflow dependencies that extend the declaration order. `items` MUST NOT be set on a `sequence`.

- **`parallel`** — Steps execute concurrently, subject only to `dependsOn` relationships. `parallelGroup` is a dependency-barrier identifier only: a `dependsOn` reference to the group waits for every member of that group. Group membership does not create implicit parallelism, extra cancellation semantics, or cross-document coordination. The construct completes when every step has terminated. `items` MUST NOT be set on a `parallel`.

- **`switch`** — Exactly one `case` whose `when` expression evaluates truthy executes its `steps`. If no `case` matches and `default` is present, `default` executes. If no `case` matches and `default` is absent, the construct emits no result. `items` MUST NOT be set on a `switch`.

- **`loop`** — Iterates over a JSON array. `items` is REQUIRED and MUST be a runtime expression that resolves to a JSON array. Each element is bound into the iteration scope and made available to expressions within `steps`. If `batchSize` is set, it MUST be a runtime expression that resolves to a positive integer; iterations are partitioned into fixed-size batches executed sequentially. `cases` and `default` MUST NOT be set on a `loop`.

- **`merge`** — Combines the outputs of multiple constructs named by `dependsOn` into a single structural result. `dependsOn` is REQUIRED on a `merge` construct and MUST name at least one upstream construct. The result is declared by a Structural Result Object whose `kind` is `merge` and whose `from` names the merge construct (see §4.5.14). `mode` is implementation-defined but, when present, MUST be a string. `items` MUST NOT be set on a `merge`.

- **`await`** — Blocks execution until its `wait` expression evaluates truthy. `wait` is REQUIRED on an `await`. Any timeout applied to `await` is an executor option, not a serialized UWS field; absent such an option, await continues until the enclosing context is canceled or completes. `cases`, `default`, and `items` MUST NOT be set on an `await`.

The constraints above apply identically to any Step Object that sets `type`. The canonical `uws.json` schema encodes them as a shared compound (`$defs/structural-type-constraints`) and applies it to workflows unconditionally and to steps via `if: { required: ["type"] }`.

| Field Name | Type | Description |
|---|---|---|
| `variables` | `Map[string, any]` | Reusable UWS-local values. Keys MUST match `^[a-zA-Z0-9._-]+$`; values are an open-shape JSON object and are not restricted by UWS. |

This object MAY be extended with specification extensions.

##### 4.5.15.2 Components Object Example

```yaml
variables:
  defaultLimit: 10
```

### 4.6 Validation

Core validation MUST check:

- `uws`, `info`, and `operations` are present.
- Version strings match the UWS 1.x pattern.
- Source description names, operation IDs, workflow IDs, step IDs, trigger IDs, and parallel groups are unique where applicable.
- Trigger output labels are unique within each trigger.
- Workflow IDs and step IDs match `^[A-Za-z0-9_-]+$` so `StructuralResult.from` remains unambiguous.
- Source descriptions are OpenAPI sources.
- Each OpenAPI-bound operation references an existing source description.
- Each OpenAPI-bound operation sets exactly one of `openapiOperationId` or `openapiOperationRef`.
- Each extension-owned operation includes `x-uws-operation-profile` with at least one non-whitespace character.
- Component variable keys match `^[a-zA-Z0-9._-]+$`.
- Dependency, operation, workflow, step, trigger route, success action, and failure action references resolve.
- Structural workflow and step types are known.
- Structural construct field constraints hold (see §4.5.6.3): `loop` requires `items`; `await` requires `wait`; `switch`, `sequence`, `parallel`, `merge`, and `await` MUST NOT set `items`; `loop` and `await` MUST NOT set `cases` or `default`.
- Each `StructuralResult` resolves `from` to a known workflow or `workflowId.stepId` pair whose `type` is structural and matches the result `kind` (see §4.5.14).
- Each trigger route `output` references a declared entry in the parent trigger's `outputs` list (see §4.5.9).
- Each trigger route `to` item references either a declared `workflowId` or a top-level `stepId` in the document's executable entry workflow (see §7.4).
- `x-*` fields are preserved as specification extensions.

Schema validation checks document shape. Semantic checks such as duplicate identifiers and reference integrity are enforced by the Go `Validate` and `ValidateResult` APIs.

## 5. Runtime Expressions

UWS uses runtime expression strings in control-flow fields (`when`, `forEach`, `wait`, `workflow`, `items`, `batchSize`), in criterion conditions, and in the string values of `outputs` maps. A runtime expression is a string whose resolved value is a JSON-compatible value determined at execution time.

Conforming implementations MUST accept the expression sources defined in §5.1, the JSON Pointer fragment form in §5.2, the output dot-walk in §5.3, and the comparison operators in §5.4. Implementations MAY support additional syntax as described in §5.5.

### 5.1 Expression Sources

Expression strings reference data through these normative sources. Each source begins with a leading `$` sigil. A dotted suffix `.<path>` is a sequence of one or more dot-separated segments, where each segment matches `[a-zA-Z0-9._-]+` and is resolved against the current value as an object property name or array index.

| Source | Meaning |
|---|---|
| `$response.statusCode` | HTTP response status code of the current operation. |
| `$response.body` | HTTP response body of the current operation. |
| `$response.body<json-pointer>` | A JSON Pointer fragment (§5.2) into the response body. |
| `$response.headers.<name>` | HTTP response header value by name. |
| `$outputs.<name>` | Same-scope output value declared by the enclosing operation, workflow, or step. |
| `$outputs.<name>.<path>` | Dot-walk into a structured output value (see §5.3). |
| `$steps.<stepId>.outputs.<name>` | Output of a named sibling step within the same workflow. |
| `$steps.<stepId>.outputs.<name>.<path>` | Dot-walk into a named sibling step's output value. |
| `$variables.<name>` | A document-scope variable declared under `variables` or `components.variables`. |
| `$variables.<name>.<path>` | Dot-walk into a structured variable value. |
| `$trigger` | The input payload delivered by the enclosing trigger. |
| `$trigger.<path>` | Dot-walk into the trigger payload. |

A resolved value MAY be of any JSON type: string, number, boolean, null, object, or array.

### 5.2 JSON Pointer Fragments

A JSON Pointer fragment `<json-pointer>` is an RFC 6901 JSON Pointer prefixed with `#`. It MAY be used as a suffix on `$response.body` and as the value of `Criterion.context` when `Criterion.type` is `jsonpath`. The pointer resolves against the value named by its prefix.

Example:

```text
$response.body#/items/0/name
```

### 5.3 Output Values and Dot-Walk

Each entry in an `outputs` map of an operation, workflow, or step is a pair `<name>: <expression>`. The `<expression>` is a string that conforms to §5.1. At execution time, each entry is evaluated and the resolved value is bound to `<name>` in the enclosing scope.

A resolved output value MAY be of any JSON type. Consumers MAY dot-walk into structured outputs:

- `$outputs.<name>` resolves to the whole bound value.
- `$outputs.<name>.<segment>` resolves to the result of applying `<segment>` as an object property name (for JSON objects) or a decimal array index (for JSON arrays).
- Subsequent segments apply recursively.

If a segment does not resolve — the property is absent from a JSON object, the index is out of bounds on a JSON array, or the segment is applied to a scalar — the expression value is `null`.

The typed shape `Map[string, string]` in the UWS document model reflects the *expression* type, not the resolved runtime type.

### 5.4 Comparison Operators

Inside `when`, `Criterion.condition` when `Criterion.type` is `simple`, and any field documented as a boolean expression, conforming implementations MUST support the comparison operators `==`, `!=`, `<`, `<=`, `>`, and `>=`.

- Operands MUST be of the same JSON type. Implementations MUST NOT perform implicit type coercion.
- `==` and `!=` are defined for strings, numbers, booleans, and `null`.
- `<`, `<=`, `>`, and `>=` are defined for strings (lexicographic) and numbers. Applying an ordering operator to booleans, `null`, objects, or arrays is an error; implementations MUST treat the comparison as failing.

### 5.5 Implementation Extensions

Richer expression features — such as boolean connectives (`&&`, `||`, `!`), arithmetic, function calls (for example `length()`), or string operations — are implementation-defined. A conforming UWS core implementation MUST NOT require such features in order to parse documents that use only §5.1–§5.4 syntax. A UWS document that depends on features beyond §5.1–§5.4 is tied to the implementation profile that defines them and SHOULD declare that dependency via an `x-uws-operation-profile` or equivalent extension marker on the relevant operation.

### 5.6 Formal Grammar

The following ABNF (RFC 5234) defines the canonical parse for UWS runtime expression strings. This grammar is normative and authoritative; where §5.1's tabular description is less specific, the grammar here applies.

```abnf
; ===== UWS 1.0 Runtime Expression Grammar (RFC 5234 ABNF) =====

expression     = condition / source-expr

condition      = source-expr SP op SP operand
operand        = source-expr / literal
op             = "==" / "!=" / "<=" / ">=" / "<" / ">"

source-expr    = response-expr
               / outputs-expr
               / steps-expr
               / variables-expr
               / trigger-expr

response-expr  = "$response.statusCode"
               / "$response.body" [ json-pointer ]
               / "$response.headers." header-name

outputs-expr   = "$outputs." name [ "." path ]
steps-expr     = "$steps." identifier ".outputs." name [ "." path ]
variables-expr = "$variables." name [ "." path ]
trigger-expr   = "$trigger" [ "." path ]

path           = segment *( "." segment )
segment        = 1*id-char
name           = 1*id-char
identifier     = 1*id-char
header-name    = 1*( ALPHA / DIGIT / "-" / "_" )
id-char        = ALPHA / DIGIT / "_" / "-"

json-pointer   = "#" *( "/" token )
token          = *( unreserved / escaped )
unreserved     = ALPHA / DIGIT / "-" / "_" / "." / "~"
escaped        = "~0" / "~1" / pct-encoded
pct-encoded    = "%" HEXDIG HEXDIG

literal        = json-string / json-number / json-bool / json-null
json-string    = DQUOTE *json-char DQUOTE
json-char      = unescaped-char / escape-seq
unescaped-char = %x20-21 / %x23-5B / %x5D-10FFFF
escape-seq     = "\" ( DQUOTE / "\" / "/" / "b" / "f" / "n" / "r" / "t"
               / "u" 4HEXDIG )
json-number    = [ "-" ] int [ frac ] [ exp ]
int            = "0" / ( %x31-39 *DIGIT )
frac           = "." 1*DIGIT
exp            = ( "e" / "E" ) [ "+" / "-" ] 1*DIGIT
json-bool      = "true" / "false"
json-null      = "null"

SP             = %x20
DQUOTE         = %x22
ALPHA          = %x41-5A / %x61-7A
DIGIT          = %x30-39
HEXDIG         = DIGIT / "A" / "B" / "C" / "D" / "E" / "F"
               / "a" / "b" / "c" / "d" / "e" / "f"
```

Interpretation notes, normative:

1. **Segment tokenization.** Path segments are dot-separated and MUST match `id-char` — that is, `^[A-Za-z0-9_-]+$`. The regex `[a-zA-Z0-9._-]+` used in §5.1 to describe the character class of output, variable, and result *names* is broader than `id-char`, but §5.6 is authoritative for expression parsing: a property name containing `.` cannot be accessed via the dotted-path suffix and MUST be reached either through a JSON Pointer fragment on `$response.body`, through a value-bearing output, or through an implementation extension (§5.5).

2. **Comparison operators.** The grammar covers a single binary comparison of the form `source-expr op operand`. Nested or parenthesised comparisons are not part of UWS core; an implementation that supports them does so under §5.5.

3. **Literals.** The `literal` production is the JSON grammar for string, number, boolean, and null values. A comparison operand that is not a `source-expr` MUST be a JSON literal.

4. **Whitespace.** A single SP (U+0020) surrounds `op` in a comparison. UWS core does not require implementations to accept additional whitespace.

5. **Operator longest-match.** `==`, `!=`, `<=`, and `>=` MUST be recognised as two-character tokens; `<` and `>` are the one-character alternatives only when the next character is not `=`.

6. **JSON Pointer escape.** Inside a `token`, `~0` represents a literal `~` and `~1` represents a literal `/`, per RFC 6901.

## 6. Specification Extensions

Specification extensions are fields whose names begin with `x-`.

UWS core does not define service types for local commands, functions, file I/O, SSH, SMTP, SQL, object storage, LLM calls, or other non-OpenAPI behavior. These capabilities can be described through `x-*` extension profiles.

Example non-normative extension:

```json
{
  "operationId": "build_email",
  "x-uws-operation-profile": "udon",
  "x-udon-runtime": {
    "type": "fnct",
    "function": "mail_raw",
    "args": {
      "subject": "Daily weather report"
    }
  }
}
```

The core specification preserves extension fields but does not validate or execute implementation-specific extension payloads.

### 6.1 Reserved Extension Prefixes

The following extension prefixes are reserved for the UWS specification and its governance. Third-party tooling MUST NOT introduce extension fields whose names begin with these prefixes.

| Prefix | Reserved for |
|---|---|
| `x-uws-` | Future evolution of UWS core. The only UWS 1.0 field under this prefix is `x-uws-operation-profile` (§4.5.4.1). Future UWS versions MAY introduce additional `x-uws-*` fields and MAY define, change, or remove their semantics. |

Implementation profiles, runtimes, and vendor-specific tooling SHOULD adopt their own unambiguous prefix, for example `x-udon-`, `x-<vendor>-`, or `x-<product>-`. An extension whose prefix is not reserved by UWS core is governed entirely by the implementation that defines it.

When two non-reserved extensions target the same object, the implementation profile named by `x-uws-operation-profile` (or its nearest enclosing equivalent) selects which extension is authoritative. UWS core does not arbitrate between conflicting third-party extensions.

Conforming tooling that encounters an unknown `x-*` field MUST preserve the field on round-trip serialization and MAY ignore the field at runtime. Tooling MUST NOT silently drop, rename, or coerce an `x-*` field whose prefix is not understood.

## 7. Execution Model

The UWS Execution Model defines the normative behavior for orchestrating workflows and executing operations. UWS adopts a bound-runtime execution model: UWS core owns structural orchestration and execution-state transitions, while an implementation-specific runtime provides the concrete execution of leaf operations and the evaluation services required by runtime expressions and iterative constructs.

### 7.1 Bound Runtime and Execution Entry Points

Execution is defined only for a document, workflow, or step that has a concrete runtime bound by the implementation. Runtime binding is execution state, not serialized UWS document data.

Before document-level execution begins, an implementation MUST validate:

1. the document as a UWS document,
2. the document as executable under the current UWS execution rules, and
3. the document's executable entrypoint requirements.

Before workflow-level or step-level execution begins, an implementation MUST validate the document as a UWS document and as executable under the current UWS execution rules.

Before trigger dispatch begins, an implementation MUST validate the document as a UWS document and as executable under the current UWS execution rules. Trigger dispatch uses the document's bound runtime and does not define a separate trigger-local runtime contract.

### 7.2 Orchestrator-Owned Semantics

The Orchestrator is the authoritative UWS core component responsible for walking the executable graph, managing execution state, and applying the semantics of the six structural constructs defined in §4.5.6.3.

At minimum, the Orchestrator MUST own:

1. indexing of declared operations, workflows, workflow steps, top-level trigger targets, and `parallelGroup` membership,
2. dependency execution across operations, workflows, workflow steps, and `parallelGroup` barriers,
3. evaluation and enforcement of structural execution attributes such as `dependsOn`, `when`, `forEach`, `items`, `batchSize`, and `wait`,
4. execution of structural workflow types (`sequence`, `parallel`, `switch`, `merge`, `loop`, `await`),
5. processing of success and failure actions, including `retry`, `goto`, and `end`,
6. resolution of declared outputs and structural results, and
7. routed execution after trigger resolution.

Trigger ingress, hosting, transport, authentication, and persistence remain implementation-specific. Once a trigger event has been accepted for dispatch, UWS core owns route resolution and routed execution semantics.

### 7.3 Runtime Responsibilities

To execute a UWS document, an implementation-specific runtime MUST provide the following capabilities to UWS core:

1. **Leaf execution.** The runtime MUST be able to execute a single Operation Object (§4.5.4), whether OpenAPI-bound or extension-owned.
2. **Expression evaluation.** The runtime MUST be able to evaluate UWS Runtime Expressions (§5) against the current execution context.
3. **Item resolution.** The runtime MUST be able to resolve the `items` or `forEach` expression used by iterative constructs into an ordered collection of items.

The runtime MAY additionally provide transport behavior, authentication handling, persistence, resumption, scheduling, worker allocation, and implementation-specific extension semantics. Those concerns are outside the serialized UWS wire format unless another specification or implementation profile defines them.

### 7.4 Execution Context

During orchestration, the implementation maintains execution context that is visible to runtime-expression evaluation and to runtime-owned leaf execution.

This execution context MAY expose:

1. a trigger context containing the active trigger identity, emitted output, output label, and payload,
2. an iteration context containing the active item, index, and batch information for iterative execution,
3. a current-execution context identifying the executable object currently being evaluated or run, and
4. a snapshot of execution records accumulated so far.

These context objects are execution state. They are not serialized UWS document fields and MUST NOT be interpreted as additions to the UWS 1.0 wire format.

### 7.5 Execution Records

An implementation MAY expose a snapshot of execution records after document execution, workflow execution, step execution, or trigger dispatch.

When execution records are exposed, each record SHOULD identify:

1. the executable object that produced the record,
2. the kind of executable object that ran,
3. the resulting status,
4. any implementation-provided result or output object, and
5. any implementation-provided error information.

The keying scheme used to address execution records is implementation-defined. UWS 1.0 does not standardize a serialized record store or a wire-format representation for execution history.

### 7.6 Trigger Dispatch

A trigger dispatch names one declared trigger, one emitted output, and one payload value.

For route resolution:

1. each `routes[].output` token is matched against the trigger's declared `outputs` list,
2. a route output token MAY identify an output by its label or by a zero-based decimal-string index into that list, and
3. a route that names an undeclared output is invalid.

Every matched route target in `routes[].to` MUST reference either:

1. a declared `workflowId`, or
2. a top-level `stepId` in the document's executable entry workflow.

Dispatch executes each routed target through the same orchestrator rules used for ordinary execution, including transitive `dependsOn` resolution, `parallelGroup` barrier handling, structural execution, and output propagation. When dispatch is active, the execution context exposes the trigger payload and emitted output through the trigger context described in §7.4.

### 7.7 Executor-Owned Options

An implementation MAY expose executor-owned execution options that affect runtime behavior without changing the UWS document itself.

In particular, the timeout used by `await` is executor-owned execution state, not a UWS 1.0 document field. If an implementation supports an await timeout, expiry of that timeout MUST terminate the active await construct with an execution error. Independent of any executor-owned timeout, context cancellation or equivalent implementation-defined cancellation MUST terminate the active await construct.

## 8. Security Considerations

UWS documents can contain request values, runtime expressions, trigger definitions, and implementation extensions. Tooling SHOULD treat documents from untrusted sources as untrusted input.

UWS core does not define credential storage. Credentials, provider defaults, and runtime-specific authentication metadata belong in implementation profiles or external configuration.

OpenAPI-bound operation security requirements are derived from referenced OpenAPI documents. UWS tooling SHOULD validate that referenced source descriptions are the intended documents before execution.
