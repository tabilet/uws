# Udon Workflow Specification v1.0.0

## Introduction

The Udon Workflow Specification (UWS) defines a standard, language-agnostic format for describing multi-service workflow orchestration. A UWS document describes a sequence of operations that may call HTTP APIs, execute shell commands, invoke functions, or connect over SSH — along with the control flow, security, and triggering mechanisms that bind them together.

UWS documents are serialized as JSON or YAML. They serve as the interchange format for the [udon](https://github.com/genelet/udon) workflow engine, complementing its native HCL-based configuration (`project.Config`). Tooling can generate, validate, and consume UWS documents independently of the udon runtime.

The specification draws inspiration from the [OpenAPI Initiative Arazzo Specification 1.0](https://spec.openapis.org/arazzo/v1.0.0), adapting its patterns for multi-protocol workflow orchestration rather than API-call sequencing alone.

The JSON Schema for UWS 1.0 documents is available at:

```
https://github.com/genelet/udon/uws/1.0/schema
```

## Definitions

### UWS Document
A self-contained description of a workflow, conforming to this specification.

### Operation
A single unit of work — an HTTP request, a shell command, an SSH remote command, or a function invocation.

### Workflow
A structural control-flow construct that composes operations or steps into parallel, switch, merge, loop, or await patterns.

### Step
A unit within a Workflow that references an operation or defines inline behavior. Steps may nest recursively.

### Trigger
An entry point that initiates workflow execution, such as a webhook endpoint or a polling schedule.

### Provider
A default service endpoint configuration (base URL, credentials) shared across operations.

### Criterion
A condition evaluated after an operation executes to determine success or failure. Supports simple expressions, regex, JSONPath, and XPath evaluation types.

### Failure Action
An action taken when an operation fails — retry the operation, jump to another workflow or step, or terminate. Failure actions may include criteria that scope them to specific failure conditions.

### Success Action
An action taken when an operation succeeds — typically jumping to a subsequent workflow or step, enabling dynamic routing.

### Runtime Expression
A string expression evaluated at execution time, used for conditional logic (`when`), iteration (`forEach`), output extraction (`outputs`), and success criteria (`successCriteria`). See [Runtime Expressions](#runtime-expressions) for the full syntax table.

### Specification Extension
A custom field prefixed with `x-` that allows tooling-specific metadata on any object.

## Versions

The `uws` field specifies which version of the UWS Specification the document conforms to. For this version, the value MUST match the pattern:

```
^1\.\d+\.\d+(-.+)?$
```

Examples of valid version strings: `1.0.0`, `1.1.0`, `1.0.0-beta.1`.

## Format

A UWS document is itself a JSON or YAML object, which may be represented in either JSON or YAML format. All field names are case-sensitive. Where `description` fields exist, they support [CommonMark](https://spec.commonmark.org/) syntax for rich text formatting.

In this specification, the schema uses the following conventions to define the data types:

| Type | Description |
|------|-------------|
| `string` | A JSON string value. |
| `integer` | A JSON integer value. |
| `boolean` | A JSON boolean value (`true` or `false`). |
| `[T]` | An array of values of type T. |
| `Map[string, T]` | A JSON object mapping string keys to values of type T. |
| `any` | Any valid JSON value. |

## Relationship to HCL

The canonical authoring format for udon workflows is [HCL](https://github.com/hashicorp/hcl) (HashiCorp Configuration Language), represented at runtime by `project.Config`. HCL provides expression evaluation, variable interpolation, function calls, conditional logic, and `for` expressions that JSON and YAML cannot express natively.

A UWS document is a **materialized snapshot** of a workflow — not its fully dynamic source definition. When converting from HCL to UWS:

- **Expression fields** (`when`, `forEach`, `wait`, `workflow`) are serialized as opaque strings. For example, the HCL expression `length(items) > 0` becomes the JSON string `"length(items) > 0"`. These strings require an HCL-compatible evaluator at runtime.
- **Structured bodies** (HCL `Body` values in `request`, `variables`, trigger `options`) are flattened to `map[string]any` via static evaluation. Dynamic expressions, variable references, and function calls within these bodies are evaluated and frozen into literal values.
- **Variable references** (`var.timeout`, `local.name`) and **interpolations** (`"Hello ${var.name}"`) are resolved to their concrete values at conversion time.

As a consequence:

| Capability | HCL (authoring) | UWS JSON/YAML (interchange) |
|---|---|---|
| Expressions | `length(items) > 0` | Preserved as opaque string |
| Variable references | `var.timeout` | Resolved to literal (e.g., `30`) |
| Function calls in bodies | `upper(name)` | Resolved to literal (e.g., `"ALICE"`) |
| String interpolation | `"api-${var.env}"` | Resolved to literal (e.g., `"api-prod"`) |
| `for` expressions | `[for x in items : x.id]` | Not representable |
| Conditional expressions | `var.env == "prod" ? 443 : 8080` | Not representable in body values |

UWS documents are designed for **interchange, validation, and documentation** — they can be consumed by any tool with a JSON parser. HCL remains the **authoring format** where the full expressiveness of the workflow logic lives.

The Go `convert` package provides helpers for JSON, YAML, and HCL conversion over UWS documents. JSON and YAML conversion preserve `x-*` extensions through the UWS JSON extension model. HCL conversion is intended for canonical UWS fields and dynamic map bodies; arbitrary `x-*` extension fields are not emitted as HCL attributes.

## Document Structure

A UWS document MUST contain the following top-level fields:

- **`uws`** — The specification version string.
- **`info`** — Metadata about the document.
- **`operations`** — At least one operation definition.

All other top-level fields are OPTIONAL.

## Schema

### UWS Object

The root object of a UWS document.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="uws-uws"></a>uws | `string` | **REQUIRED**. The version of the UWS Specification this document conforms to. MUST match the pattern `^1\.\d+\.\d+(-.+)?$`. |
| <a name="uws-info"></a>info | [Info Object](#info-object) | **REQUIRED**. Metadata about the workflow document. |
| <a name="uws-sourceDescriptions"></a>sourceDescriptions | \[[Source Description Object](#source-description-object)\] | A list of source documents (typically OpenAPI specifications) that operations in this document are derived from. Each source description MUST have a unique `name`. |
| <a name="uws-provider"></a>provider | [Provider Object](#provider-object) | A default provider configuration applied to all operations that do not specify their own. |
| <a name="uws-variables"></a>variables | `Map[string, any]` | Global variables available to all operations and expressions at runtime. |
| <a name="uws-operations"></a>operations | \[[Operation Object](#operation-object)\] | **REQUIRED**. An ordered list of operations. MUST contain at least one item. Each operation MUST have a unique `operationId`. |
| <a name="uws-workflows"></a>workflows | \[[Workflow Object](#workflow-object)\] | Structural workflow constructs that compose operations into higher-level control flows. Each workflow MUST have a unique `workflowId`. |
| <a name="uws-triggers"></a>triggers | \[[Trigger Object](#trigger-object)\] | Entry points that initiate workflow execution. Each trigger MUST have a unique `triggerId`. |
| <a name="uws-security"></a>security | \[[Security Requirement Object](#security-requirement-object)\] | Global security requirements applied to all HTTP operations by default. Individual operations may override this with their own `security` field. |
| <a name="uws-results"></a>results | \[[Structural Result Object](#structural-result-object)\] | Output declarations from structural constructs. |
| <a name="uws-components"></a>components | [Components Object](#components-object) | Reusable object definitions scoped to this document. |

This object MAY be extended with [Specification Extensions](#specification-extensions).

#### UWS Object Example

```json
{
    "uws": "1.0.0",
    "info": {
        "title": "Pet Store Workflow",
        "version": "1.0.0"
    },
    "sourceDescriptions": [
        {
            "name": "petstore_api",
            "url": "./petstore.yaml",
            "type": "openapi"
        }
    ],
    "provider": {
        "name": "petstore",
        "serverUrl": "https://api.petstore.example.com"
    },
    "variables": {
        "env": "production",
        "timeout": 30
    },
    "operations": [
        {
            "operationId": "list_pets",
            "serviceType": "http",
            "sourceDescription": "petstore_api",
            "method": "GET",
            "path": "/pets"
        }
    ]
}
```

### Source Description Object

Identifies a source document that operations in this UWS document are derived from. Typically this is an OpenAPI specification, but it may also be another Arazzo or UWS document.

Each source description is referenced by its `name` from the `sourceDescription` field on individual [Operation Objects](#operation-object). This establishes provenance — linking each operation back to the API specification it was generated from.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="sourcedescription-name"></a>name | `string` | **REQUIRED**. A unique name for this source description. MUST match the pattern `^[A-Za-z0-9_\-]+$`. |
| <a name="sourcedescription-url"></a>url | `string` | **REQUIRED**. A URL or relative path to the source document. |
| <a name="sourcedescription-type"></a>type | `string` | The type of source document. MUST be one of: `openapi`, `arazzo`. |

This object MAY be extended with [Specification Extensions](#specification-extensions).

#### Source Description Object Example

```json
{
    "name": "petstore_api",
    "url": "./petstore.yaml",
    "type": "openapi"
}
```

#### Multiple Source Descriptions Example

```json
{
    "sourceDescriptions": [
        {
            "name": "petstore_api",
            "url": "./petstore.yaml",
            "type": "openapi"
        },
        {
            "name": "payments_api",
            "url": "https://api.payments.example.com/openapi.json",
            "type": "openapi"
        }
    ]
}
```

### Operation Object

Describes a single unit of work. Every operation has a `serviceType` that determines which fields are required and how the operation is executed at runtime.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="operation-operationId"></a>operationId | `string` | **REQUIRED**. A unique identifier for this operation within the document. |
| <a name="operation-serviceType"></a>serviceType | `string` | **REQUIRED**. The protocol or execution model. See [Appendix A: Service Types](#appendix-a-service-types) for valid values. |
| <a name="operation-sourceDescription"></a>sourceDescription | `string` | The `name` of a [Source Description Object](#source-description-object) that this operation is derived from. Links the operation back to its source API specification. |
| <a name="operation-description"></a>description | `string` | A human-readable description of what the operation does. |
| <a name="operation-isJson"></a>isJson | `boolean` | Whether the response body should be parsed as JSON. |
| <a name="operation-host"></a>host | `string` | The remote host address (for SSH operations or host overrides). |
| <a name="operation-method"></a>method | `string` | The HTTP method. **REQUIRED** when `serviceType` is `http`. MUST be one of: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, `OPTIONS`, `TRACE`. |
| <a name="operation-path"></a>path | `string` | The URL path (for HTTP operations) or file/command path. |
| <a name="operation-provider"></a>provider | [Provider Object](#provider-object) | An operation-level provider that overrides the document-level provider. |
| <a name="operation-request"></a>request | `Map[string, any]` | Request parameters or body as key-value pairs. |
| <a name="operation-security"></a>security | \[[Security Requirement Object](#security-requirement-object)\] | Operation-level security requirements. When present, these override the document-level `security` for this operation. |
| <a name="operation-queryPars"></a>queryPars | [Param Schema Object](#param-schema-object) | Schema for URL query parameters. |
| <a name="operation-pathPars"></a>pathPars | [Param Schema Object](#param-schema-object) | Schema for URL path parameters. |
| <a name="operation-headerPars"></a>headerPars | [Param Schema Object](#param-schema-object) | Schema for HTTP request headers. |
| <a name="operation-cookiePars"></a>cookiePars | [Param Schema Object](#param-schema-object) | Schema for HTTP cookies. |
| <a name="operation-payloadPars"></a>payloadPars | [Param Schema Object](#param-schema-object) | Schema for the request payload body. |
| <a name="operation-payloadRequired"></a>payloadRequired | `boolean` | Whether the request payload is mandatory. |
| <a name="operation-requestMediaType"></a>requestMediaType | `string` | The media type of the request body (e.g., `application/json`). |
| <a name="operation-responseMediaType"></a>responseMediaType | `string` | The expected media type of the response body. |
| <a name="operation-responseBody"></a>responseBody | [Param Schema Object](#param-schema-object) | Schema describing the response payload. |
| <a name="operation-responseHeaders"></a>responseHeaders | [Param Schema Object](#param-schema-object) | Schema describing the response headers. |
| <a name="operation-responseStatusCode"></a>responseStatusCode | `integer` | The expected HTTP response status code. |
| <a name="operation-command"></a>command | `string` | The command to execute. **REQUIRED** when `serviceType` is `ssh` or `cmd`. |
| <a name="operation-arguments"></a>arguments | `[any]` | Positional arguments passed to the command or function. |
| <a name="operation-workingDir"></a>workingDir | `string` | The working directory for `cmd` operations. |
| <a name="operation-function"></a>function | `string` | The fully-qualified function name. **REQUIRED** when `serviceType` is `fnct`. |
| <a name="operation-dependsOn"></a>dependsOn | `[string]` | A list of `operationId` values or parallel group names that must complete before this operation executes. |
| <a name="operation-when"></a>when | `string` | A runtime expression that must evaluate to true for this operation to execute. If the expression evaluates to false, the operation is skipped. |
| <a name="operation-forEach"></a>forEach | `string` | A runtime expression that resolves to an iterable. The operation executes once for each item. |
| <a name="operation-wait"></a>wait | `string` | A duration or expression specifying how long to wait before executing this operation. |
| <a name="operation-workflow"></a>workflow | `string` | A reference to a sub-workflow to execute as part of this operation. |
| <a name="operation-parallelGroup"></a>parallelGroup | `string` | Assigns this operation to a named parallel group. Operations in the same group execute concurrently. |
| <a name="operation-successCriteria"></a>successCriteria | \[[Criterion Object](#criterion-object)\] | Conditions that determine whether the operation succeeded. When present, the runtime evaluates each criterion after execution; if any fails, the operation is considered failed. |
| <a name="operation-onFailure"></a>onFailure | \[[Failure Action Object](#failure-action-object)\] | Actions to take when the operation fails (or when `successCriteria` are not met). Evaluated in order; the first matching action is executed. |
| <a name="operation-onSuccess"></a>onSuccess | \[[Success Action Object](#success-action-object)\] | Actions to take when the operation succeeds. Enables dynamic routing after successful execution. |
| <a name="operation-outputs"></a>outputs | `Map[string, string]` | Named output values extracted via runtime expressions (e.g., `"petList": "$response.body.items"`). |

This object MAY be extended with [Specification Extensions](#specification-extensions).

#### Service Type Requirements

The following fields are conditionally required based on the value of `serviceType`:

| serviceType | Required Field |
|-------------|----------------|
| `http` | `method` |
| `ssh` | `command` |
| `cmd` | `command` |
| `fnct` | `function` |

#### HTTP Operation Example

```json
{
    "operationId": "list_pets",
    "serviceType": "http",
    "sourceDescription": "petstore_api",
    "method": "GET",
    "path": "/pets",
    "isJson": true,
    "queryPars": {
        "type": "object",
        "properties": {
            "limit": { "type": "integer" },
            "status": { "type": "string" }
        }
    },
    "request": {
        "limit": 10,
        "status": "available"
    },
    "responseStatusCode": 200,
    "outputs": {
        "petList": "$response.body.items"
    }
}
```

#### Command Operation Example

```json
{
    "operationId": "check_disk",
    "serviceType": "cmd",
    "command": "df",
    "arguments": ["-h", "/"],
    "workingDir": "/tmp"
}
```

#### Function Operation Example

```json
{
    "operationId": "compute_hash",
    "serviceType": "fnct",
    "function": "crypto.SHA256",
    "arguments": ["input_data"],
    "dependsOn": ["list_pets"]
}
```

#### Conditional Operation Example

```json
{
    "operationId": "create_pet",
    "serviceType": "http",
    "method": "POST",
    "path": "/pets",
    "isJson": true,
    "payloadRequired": true,
    "dependsOn": ["list_pets"],
    "when": "length(list_pets.received_body.items) < 100",
    "request": {
        "name": "Buddy",
        "tag": "dog"
    }
}
```

#### Operation with Success Criteria and Failure Actions

```json
{
    "operationId": "call_api",
    "serviceType": "http",
    "method": "POST",
    "path": "/api/resource",
    "successCriteria": [
        { "condition": "$statusCode == 200" },
        { "condition": "^\\{", "type": "regex", "context": "$response.body" }
    ],
    "onFailure": [
        {
            "name": "retry_on_503",
            "type": "retry",
            "retryAfter": 5,
            "retryLimit": 3,
            "criteria": [
                { "condition": "$statusCode == 503" }
            ]
        },
        {
            "name": "fallback",
            "type": "goto",
            "workflowId": "error_handler"
        }
    ],
    "onSuccess": [
        {
            "name": "continue_to_validation",
            "type": "goto",
            "workflowId": "validate_response"
        }
    ],
    "outputs": {
        "resourceId": "$response.body#/id"
    }
}
```

### Criterion Object

Describes a condition that determines whether an operation succeeded or whether an action should be applied. Criteria are protocol-agnostic — the same model works for HTTP status codes, command exit codes, and function return values.

When `type` is `simple` (the default), the `condition` is evaluated as a runtime expression that must be truthy. When `type` is `regex`, `jsonpath`, or `xpath`, the `condition` is applied against the value resolved by `context`.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="criterion-condition"></a>condition | `string` | **REQUIRED**. The condition to evaluate. For `simple` type, a [runtime expression](#runtime-expressions). For `regex` type, a regular expression pattern. For `jsonpath`/`xpath`, the respective query. |
| <a name="criterion-type"></a>type | `string` | How to evaluate the condition. MUST be one of: `simple`, `regex`, `jsonpath`, `xpath`. Defaults to `simple`. |
| <a name="criterion-context"></a>context | `string` | A runtime expression resolving to the value against which the `condition` is applied. REQUIRED when `type` is `regex`, `jsonpath`, or `xpath`. |

This object MAY be extended with [Specification Extensions](#specification-extensions).

#### Criterion Examples

HTTP status check (simple):

```json
{ "condition": "$statusCode == 200" }
```

Command exit code (simple):

```json
{ "condition": "$exitCode == 0" }
```

Function return value (simple):

```json
{ "condition": "$returnValue != null" }
```

Response body regex match:

```json
{
    "condition": "^/dev/",
    "type": "regex",
    "context": "$response.body"
}
```

### Failure Action Object

Describes what to do when an operation fails or its `successCriteria` are not met. Actions are evaluated in order; the first one whose `criteria` match (or that has no `criteria`) is executed.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="failureaction-name"></a>name | `string` | **REQUIRED**. A unique name for this action. |
| <a name="failureaction-type"></a>type | `string` | **REQUIRED**. The action type. MUST be one of: `end` (terminate the workflow), `goto` (jump to another workflow or step), `retry` (re-execute the operation). |
| <a name="failureaction-workflowId"></a>workflowId | `string` | The target workflow for `goto` actions. At least one of `workflowId` or `stepId` is REQUIRED when `type` is `goto`. |
| <a name="failureaction-stepId"></a>stepId | `string` | The target step for `goto` actions. |
| <a name="failureaction-retryAfter"></a>retryAfter | `number` | Seconds to wait before retrying. Only meaningful when `type` is `retry`. |
| <a name="failureaction-retryLimit"></a>retryLimit | `integer` | **REQUIRED when `type` is `retry`**. Maximum number of retry attempts. MUST be greater than 0. |
| <a name="failureaction-criteria"></a>criteria | \[[Criterion Object](#criterion-object)\] | Conditions that must all match for this action to apply. If omitted, the action always applies (acts as a default/catch-all). |

This object MAY be extended with [Specification Extensions](#specification-extensions).

#### Failure Action Example — Retry with Criteria

```json
{
    "name": "retry_on_timeout",
    "type": "retry",
    "retryAfter": 5,
    "retryLimit": 3,
    "criteria": [
        { "condition": "$statusCode == 503" }
    ]
}
```

#### Failure Action Example — Goto Fallback

```json
{
    "name": "fallback_handler",
    "type": "goto",
    "workflowId": "error_recovery"
}
```

### Success Action Object

Describes what to do when an operation succeeds. Success actions enable dynamic routing — for example, jumping to a validation workflow after creating a resource.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="successaction-name"></a>name | `string` | **REQUIRED**. A unique name for this action. |
| <a name="successaction-type"></a>type | `string` | **REQUIRED**. The action type. MUST be one of: `end` (terminate the workflow), `goto` (jump to another workflow or step). |
| <a name="successaction-workflowId"></a>workflowId | `string` | The target workflow for `goto` actions. At least one of `workflowId` or `stepId` is REQUIRED when `type` is `goto`. |
| <a name="successaction-stepId"></a>stepId | `string` | The target step for `goto` actions. |
| <a name="successaction-criteria"></a>criteria | \[[Criterion Object](#criterion-object)\] | Conditions that must all match for this action to apply. If omitted, the action always applies. |

This object MAY be extended with [Specification Extensions](#specification-extensions).

#### Success Action Example

```json
{
    "name": "goto_validation",
    "type": "goto",
    "workflowId": "validate_resource",
    "criteria": [
        { "condition": "$response.body#/id != null" }
    ]
}
```

### Param Schema Object

Describes the schema of a parameter, payload, or response. This object is recursive — `properties` values, `items`, and composition keywords (`allOf`, `oneOf`, `anyOf`) are themselves Param Schema Objects.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="paramschema-type"></a>type | `string` | The data type (e.g., `object`, `string`, `integer`, `array`, `boolean`, `number`). |
| <a name="paramschema-format"></a>format | `string` | An optional format qualifier (e.g., `date-time`, `uuid`, `int64`). |
| <a name="paramschema-ref"></a>$ref | `string` | A JSON Reference to another schema definition. |
| <a name="paramschema-properties"></a>properties | `Map[string, `[Param Schema Object](#param-schema-object)`]` | Property definitions when `type` is `object`. |
| <a name="paramschema-required"></a>required | `[string]` | A list of required property names when `type` is `object`. |
| <a name="paramschema-items"></a>items | [Param Schema Object](#param-schema-object) | The schema for array items when `type` is `array`. |
| <a name="paramschema-allOf"></a>allOf | \[[Param Schema Object](#param-schema-object)\] | An array of schemas; the value MUST validate against all of them. |
| <a name="paramschema-oneOf"></a>oneOf | \[[Param Schema Object](#param-schema-object)\] | An array of schemas; the value MUST validate against exactly one. |
| <a name="paramschema-anyOf"></a>anyOf | \[[Param Schema Object](#param-schema-object)\] | An array of schemas; the value MUST validate against at least one. |

This object MAY be extended with [Specification Extensions](#specification-extensions).

#### Param Schema Object Example

```json
{
    "type": "object",
    "properties": {
        "items": {
            "type": "array",
            "items": {
                "type": "object",
                "properties": {
                    "id": { "type": "integer" },
                    "name": { "type": "string" }
                }
            }
        },
        "total": { "type": "integer" }
    },
    "required": ["items"]
}
```

### Security Requirement Object

Describes a security mechanism applied to operations. When listed at the document level in `security`, it applies to all HTTP operations by default. Operations MAY override with their own `security` field.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="securityrequirement-name"></a>name | `string` | A name for this security requirement. |

### Workflow Object

Defines a structural control-flow construct that composes operations or steps into higher-level execution patterns.

#### Fixed Fields

| Field Name | Type | Description |
|------------|------|-------------|
| <a name="workflow-workflowId"></a>workflowId | `string` | **REQUIRED**. A unique identifier for this workflow within the document. |
| <a name="workflow-type"></a>type | `string` | **REQUIRED**. The structural construct type. MUST be one of: `parallel`, `switch`, `merge`, `loop`, `await`. |
| <a name="workflow-description"></a>description | `string` | A human-readable description of the workflow's purpose. |
| <a name="workflow-inputs"></a>inputs | [Param Schema Object](#param-schema-object) | A JSON Schema defining the workflow's input parameters. Makes the workflow self-documenting and composable — callers know exactly what data to provide. |
| <a name="workflow-dependsOn"></a>dependsOn | `[string]` | Operation or workflow IDs that must complete before this workflow executes. |
| <a name="workflow-when"></a>when | `string` | A runtime expression controlling conditional execution. |
| <a name="workflow-forEach"></a>forEach | `string` | A runtime expression for iteration. |
| <a name="workflow-wait"></a>wait | `string` | A duration or expression specifying a wait before execution. |
| <a name="workflow-items"></a>items | `string` | A runtime expression resolving to the collection to iterate or switch on. Used by `loop` and `switch` types. |
| <a name="workflow-mode"></a>mode | `string` | Execution mode qualifier for `switch` or `merge` types. |
| <a name="workflow-batchSize"></a>batchSize | `string` | The number of items to process per batch in `loop` types. |
| <a name="workflow-steps"></a>steps | \[[Step Object](#step-object)\] | The steps within this workflow. Used by `parallel`, `loop`, and `await` types. |
| <a name="workflow-cases"></a>cases | \[[Case Object](#case-object)\] | The branches for `switch` type workflows. |
| <a name="workflow-default"></a>default | \[[Step Object](#step-object)\] | Default steps executed when no `case` matches in a `switch` workflow. |
| <a name="workflow-outputs"></a>outputs | `Map[string, string]` | Named output values produced by this workflow, expressed as runtime expressions (e.g., `"allResults": "$steps.*.response.body"`). Makes workflows composable by declaring what they produce. |

This object MAY be extended with [Specification Extensions](#specification-extensions).

#### Parallel Workflow Example

```json
{
    "workflowId": "parallel_checks",
    "type": "parallel",
    "description": "Run validation and logging in parallel after pet creation",
    "inputs": {
        "type": "object",
        "properties": {
            "petId": { "type": "integer" }
        },
        "required": ["petId"]
    },
    "dependsOn": ["create_pet"],
    "steps": [
        {
            "stepId": "validate_pet",
            "type": "http",
            "description": "Validate the newly created pet record",
            "operationRef": "create_pet",
            "body": {
                "path": "/pets/validate"
            },
            "outputs": {
                "isValid": "$response.body.valid"
            }
        },
        {
            "stepId": "log_action",
            "type": "cmd",
            "description": "Log the pet creation event",
            "body": {
                "command": "echo",
                "arguments": ["Pet created"]
            }
        }
    ],
    "outputs": {
        "validationResult": "$steps.validate_pet.outputs.isValid"
    }
}
```

#### Switch Workflow Example

```json
{
    "workflowId": "route_by_type",
    "type": "switch",
    "items": "$operations.list_pets.response.body",
    "cases": [
        {
            "name": "dog",
            "when": "item.type == 'dog'",
            "steps": [
                {
                    "stepId": "process_dog",
                    "type": "http",
                    "operationRef": "create_pet"
                }
            ]
        },
        {
            "name": "cat",
            "when": "item.type == 'cat'"
        }
    ],
    "default": [
        {
            "stepId": "log_unknown",
            "type": "cmd",
            "body": {
                "command": "echo",
                "arguments": ["Unknown animal type"]
            }
        }
    ]
}
```

## Runtime Expressions

Runtime expressions are strings used in `when`, `forEach`, `outputs`, `successCriteria`, and other fields to reference values produced during workflow execution. UWS extends the [Arazzo 1.0 runtime expression](https://spec.openapis.org/arazzo/v1.0.0#runtime-expressions) syntax for multi-protocol support.

### Expression Syntax

| Expression | Scope | Description |
|---|---|---|
| `$statusCode` | HTTP operations | The HTTP response status code (integer). |
| `$exitCode` | cmd/ssh operations | The process exit code (integer). |
| `$returnValue` | fnct operations | The function return value. |
| `$response.body` | All operations | The raw response body as a string. |
| `$response.body#/json/pointer` | All operations | A value within the response body, resolved via [JSON Pointer](https://www.rfc-editor.org/rfc/rfc6901) (e.g., `$response.body#/items/0/id`). |
| `$response.header.{name}` | HTTP operations | The value of a response header. |
| `$request.query.{name}` | HTTP operations | The value of a request query parameter. |
| `$request.body#/json/pointer` | HTTP operations | A value within the request body. |
| `$inputs.{name}` | Workflows | A workflow input parameter. |
| `$steps.{stepId}.outputs.{name}` | Workflows | An output from a step within the same workflow. |
| `$operations.{operationId}.outputs.{name}` | Global | An output from a named operation. |
| `$variables.{name}` | Global | A global variable value. |

### JSON Pointer Syntax

UWS uses JSON Pointers (`#/path/to/field`) rather than dot notation to navigate into response and request bodies. This aligns with [RFC 6901](https://www.rfc-editor.org/rfc/rfc6901) and JSON Schema `$ref` conventions.

Examples:

```
$response.body#/items           → the "items" array
$response.body#/items/0/id      → the "id" of the first item
$response.body#/metadata/count  → nested field access
```

### Expression Context

In `successCriteria`, the expression context depends on the operation's `serviceType`:

| serviceType | Available Expressions |
|---|---|
| `http` | `$statusCode`, `$response.body`, `$response.header.*`, `$request.*` |
| `cmd`, `ssh` | `$exitCode`, `$response.body` |
| `fnct` | `$returnValue` |

All service types additionally have access to `$variables.*`, `$operations.*`, and `$inputs.*` (when within a workflow).

## Specification Extensions

All objects in a UWS document MAY include additional fields beyond those defined in this specification, provided the field names begin with `x-`. These are called Specification Extensions.

Extension field names MUST match the pattern `^x-`. The value MAY be any valid JSON value (`null`, a primitive, an array, or an object).

Extensions are preserved during serialization and deserialization but have no defined behavior in the specification itself. They are intended for tooling-specific metadata — for example:

```json
{
    "uws": "1.0.0",
    "info": {
        "title": "My Workflow",
        "version": "1.0.0",
        "x-owner": "platform-team"
    },
    "operations": [
        {
            "operationId": "fetch_data",
            "serviceType": "http",
            "method": "GET",
            "path": "/data",
            "x-retry-count": 3,
            "x-timeout": "30s"
        }
    ],
    "x-generator": "udon-cli"
}
```

## Appendix A: Service Types

The `serviceType` field on an [Operation Object](#operation-object) determines the protocol or execution model. The following values are defined:

### Core Service Types

| Value | Description | Required Fields |
|-------|-------------|-----------------|
| `http` | HTTP/HTTPS API call | `method` |
| `ssh` | Remote command execution over SSH | `command` |
| `cmd` | Local command/process execution | `command` |
| `fnct` | In-process function invocation | `function` |

### Extended Service Types

The following service types are defined for forward compatibility. Their detailed behavior is implementation-defined.

| Value | Description |
|-------|-------------|
| `fileio` | File system read/write operations |
| `ftp` | FTP file transfer |
| `sftp` | SFTP (SSH-based) file transfer |
| `scp` | SCP file copy |
| `smtp` | Email sending via SMTP |
| `dns` | DNS lookup operations |
| `sql` | SQL database queries |
| `s3` | S3-compatible object storage |
| `docker` | Docker container operations |
| `ldap` | LDAP directory operations |
| `llm` | Large language model inference |
| `mcp` | Model Context Protocol tool invocation |

### IO Service Types

| Value | Description |
|-------|-------------|
| `ioreadcloser` | Operations on `io.ReadCloser` streams |
| `ioreadwritecloser` | Operations on `io.ReadWriteCloser` streams |
| `iowriter` | Operations on `io.Writer` streams |
